What is Appium?

Appium is an open-source mobile automation framework that uses the WebDriver protocol — it works with native apps, hybrid apps, and mobile web (Chrome on Android, Safari on iOS).

  • Appium 2 — modular architecture with pluggable drivers
  • Drivers: uiautomator2 (Android) · xcuitest (iOS)
  • Client in this project: WebdriverIO 9 + Mocha + TypeScript
  • TestFlow via mobile browser — mirror of Cypress, Playwright, and PyTest suites
  • Failure screenshots, Allure Report, and visual baselines
This project uses mobile web — Appium controls Chrome/Safari on an emulator or real device, not a native APK/IPA. The same TestFlow app runs at http://10.0.2.2:5050 (Android) or http://127.0.0.1:5050 (iOS).

Why WebdriverIO?

Native Appium integration

@wdio/appium-service starts the Appium server automatically in the runner.

expect-webdriverio

Auto-wait assertions: toBeDisplayed(), toHaveUrl(), toHaveText().

TypeScript first

Auto-compile via ts-node — integrated WDIO and Mocha types.

Allure + spec reporter

Screenshots, URL, and env vars attached automatically on failure.

Installation — testflow-appium

  1. Clone the repository
    git clone https://github.com/lflucasferreira/testflow-appium.git
    cd testflow-appium
  2. Install dependencies
    npm install
  3. Install Appium drivers
    npm run appium:drivers
  4. Configure environment variables
    cp .env.example .env
  5. Validate the environment
    npm run appium:doctor
    npm run check:testflow
  6. Run tests
    npm run test:api (no emulator)
    npm run test:android (Android emulator)
Appium 2 is installed locally via npm install — no need for npm install -g appium. The WDIO service starts the server automatically.

Mobile URL mapping

Emulators cannot reach localhost the same way desktop browsers do. The project resolves the base URL automatically in support/config.ts.

PlatformDefault URLOverride
Android emulatorhttp://10.0.2.2:5050ANDROID_BASE_URL
iOS simulatorhttp://127.0.0.1:5050IOS_BASE_URL
CI (adb reverse)http://127.0.0.1:5050BASE_URL
Desktop / API onlyhttp://localhost:5050API_BASE_URL
 
// support/config.ts
export function getBaseUrl(): string {
  const platform = detectPlatform()
  if (platform === 'android') return ANDROID_LOOPBACK  // 10.0.2.2:5050
  if (platform === 'ios') return IOS_LOOPBACK          // 127.0.0.1:5050
  return DEFAULT_BASE_URL                              // localhost:5050
}

export function appPath(path: string): string {
  return `${getBaseUrl()}${path}`
}

Selector helpers — support/selectors.ts

export const byTestId = (id: string) => `[data-testid="${id}"]`

export async function $(testId: string) {
  return browser.$(byTestId(testId))
}

export async function $$(testIdPrefix: string) {
  return browser.$$(`[data-testid^="${testIdPrefix}"]`)
}

export async function waitForTestId(testId: string, timeout?) {
  const el = await $(testId)
  await el.waitForExist({ timeout })
  await el.waitForDisplayed({ timeout })
  return el
}

// support/selectors.ts

  `[data-testid="${id}"]`

export async function clickTestId(testId: string) {
  const el = await waitForTestId(testId)
  await el.click()
}

export async function fillTestId(testId: string, value: string) {
  const el = await waitForTestId(testId)
  await el.clearValue()
  await el.setValue(value)

export async function loginViaApi(): Promise<AuthSession> {
  const session = await fetchAuthToken()
  await seedSessionOnOrigin(() => injectAuthSession(session))
  await openAppPath('/web/dashboard.html')
  await waitForTestId('page-dashboard')
  return session
}

seedSessionOnOrigin opens /web/index.html first to ensure sessionStorage is on the correct origin before injecting the token.

Page Object Model

// pages/BasePage.ts
export class BasePage {
  protected testId(id: string) { return $(id) }

  async open(path: string) {
    await openAppPath(path)
    return this
  }

  async waitForPage(testId: string) {
    await waitForTestId(testId)
    return this
  }
}

Each Page Object encapsulates locators (testId()), fluent actions (loginWith(), search()), and assertions (shouldBeLoaded(), shouldShowError()).

fluent API return this data-testid only

LoginPage — complete example

export class LoginPage extends BasePage {
  emailInput()    { return this.testId('login-email') }
  passwordInput() { return this.testId('login-password') }
  submitBtn()     { return this.testId('login-submit') }
  resultMsg()     { return this.testId('login-result') }

  async visit() { await this.open('/web/login.html'); return this }

  async loginWith(email: string, password: string) {
    await this.fillEmail(email)
    await this.fillPassword(password)
    await this.submit()
    return this
  }

  async shouldRedirectToDashboard() {
    await this.waitForPage('page-dashboard')
    await expect(browser).toHaveUrl(expect.stringContaining('/web/dashboard.html'))
    return this
  }
}

Best practices

  • Page Objects in pages/ — isolated locators, actions, and assertions
  • data-testid via byTestId() — same convention as testflow-cypress/playwright
  • loginViaApi() in beforeEach — feature specs without repeating UI login
  • seedSessionOnOrigin() — injects token on the correct origin before navigating
  • getBaseUrl() — resolves loopback automatically per platform
  • hideKeyboardIfShown() — close keyboard before clicking buttons on Android
  • scrollIntoView() before gestures — element visible in viewport
  • maxInstances: 1 — one device at a time, avoids session conflicts
  • Tags @smoke / @regression / @appium — filter in CI
  • tc(TC.…) — traceable [TC-xxxx] IDs in test titles
  • validateSchema — declarative REST response types
  • assertScreenshotMatches — visual baselines with configurable threshold
  • Screenshots in afterTest — automatic failure diagnostics
  • npm run test:api first — validates TestFlow without emulator
  • npm run appium:doctor — validate drivers before first mobile run
  • One scenario per it() — easy to diagnose failures in Allure
  • visitWithToken in smoke — shared token via before
  • JSON fixtures in fixtures/ — deterministic valid/invalid credentials
