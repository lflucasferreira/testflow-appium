O que é o Appium?

O Appium é um framework open-source de automação mobile que usa o protocolo WebDriver — funciona com apps nativos, híbridos e mobile web (Chrome no Android, Safari no iOS).

  • Appium 2 — arquitetura modular com drivers plugáveis
  • Drivers: uiautomator2 (Android) · xcuitest (iOS)
  • Cliente neste projeto: WebdriverIO 9 + Mocha + TypeScript
  • TestFlow via browser mobile — espelho dos suites Cypress, Playwright e PyTest
  • Screenshots em falha, Allure Report e baselines visuais
Este projeto usa mobile web — o Appium controla Chrome/Safari no emulador ou dispositivo real, não um APK/IPA nativo. A mesma app TestFlow roda em http://10.0.2.2:5050 (Android) ou http://127.0.0.1:5050 (iOS).

@wdio/appium-service sobe o servidor Appium automaticamente no runner.

TypeScript first

Auto-compile via ts-node — tipos WDIO e Mocha integrados.

Allure + spec reporter

Screenshots, URL e env vars anexados automaticamente em falha.

    cd testflow-appium
  2. Instale dependências
    npm install
  3. Instale drivers Appium
    npm run appium:drivers
  4. Configure variáveis de ambiente
    cp .env.example .env
  5. Valide o ambiente
    npm run appium:doctor
    npm run check:testflow
  6. Execute os testes

O Appium 2 é instalado localmente via npm install — não é necessário npm install -g appium. O serviço WDIO sobe o servidor automaticamente.

Mobile URL mapping

Emuladores não alcançam localhost da mesma forma que browsers desktop. O projeto resolve a base URL automaticamente em support/config.ts.

PlataformaURL padrãoOverride
Android emulatorhttp://10.0.2.2:5050ANDROID_BASE_URL
iOS simulatorhttp://127.0.0.1:5050IOS_BASE_URL
CI (adb reverse)http://127.0.0.1:5050BASE_URL
Desktop / API onlyhttp://localhost:5050API_BASE_URL
 
// support/config.ts
export function getBaseUrl(): string {
  const platform = detectPlatform()
  if (platform === 'android') return ANDROID_LOOPBACK  // 10.0.2.2:5050
  if (platform === 'ios') return IOS_LOOPBACK          // 127.0.0.1:5050
  return DEFAULT_BASE_URL                              // localhost:5050
}

export function appPath(path: string): string {
  return `${getBaseUrl()}${path}`
}

Helpers de seletores — support/selectors.ts

export const byTestId = (id: string) => `[data-testid="${id}"]`

export async function $(testId: string) {
  return browser.$(byTestId(testId))
}

export async function $$(testIdPrefix: string) {
  return browser.$$(`[data-testid^="${testIdPrefix}"]`)
}

export async function waitForTestId(testId: string, timeout?) {
  const el = await $(testId)
  await el.waitForExist({ timeout })
  await el.waitForDisplayed({ timeout })
  return el
}

export async function loginViaApi(): Promise<AuthSession> {
  const session = await fetchAuthToken()
  await seedSessionOnOrigin(() => injectAuthSession(session))
  await openAppPath('/web/dashboard.html')
  await waitForTestId('page-dashboard')
  return session
}

seedSessionOnOrigin abre /web/index.html primeiro para garantir que o sessionStorage está no origin correto antes de injetar o token.

Page Object Model

// pages/BasePage.ts
export class BasePage {
  protected testId(id: string) { return $(id) }

  async open(path: string) {
    await openAppPath(path)
    return this
  }

  async waitForPage(testId: string) {
    await waitForTestId(testId)
    return this
  }
}

Cada Page Object encapsula locators (testId()), ações fluentes (loginWith(), search()) e assertions (shouldBeLoaded(), shouldShowError()).

fluent API return this data-testid only

LoginPage — exemplo completo

export class LoginPage extends BasePage {
  emailInput()    { return this.testId('login-email') }
  passwordInput() { return this.testId('login-password') }
  submitBtn()     { return this.testId('login-submit') }
  resultMsg()     { return this.testId('login-result') }

  async visit() { await this.open('/web/login.html'); return this }

  async loginWith(email: string, password: string) {
    await this.fillEmail(email)
    await this.fillPassword(password)
    await this.submit()
    return this
  }

  async shouldRedirectToDashboard() {
    await this.waitForPage('page-dashboard')
    await expect(browser).toHaveUrl(expect.stringContaining('/web/dashboard.html'))
    return this
  }
}

Boas práticas

  • Page Objects em pages/ — locators, ações e assertions isolados
  • data-testid via byTestId() — mesma convenção do testflow-cypress/playwright
  • loginViaApi() no beforeEach — specs de feature sem repetir login UI
  • seedSessionOnOrigin() — injeta token no origin correto antes de navegar
  • getBaseUrl() — resolve loopback automaticamente por plataforma
  • hideKeyboardIfShown() — fecha teclado antes de clicar em botões Android
  • scrollIntoView() antes de gestos — elemento visível no viewport
  • maxInstances: 1 — um device por vez, evita conflito de sessão
  • Tags @smoke / @regression / @appium — filtrar no CI
  • tc(TC.…) — IDs rastreáveis [TC-xxxx] no título do teste
  • validateSchema — tipos da resposta REST de forma declarativa
  • assertScreenshotMatches — baselines visuais com threshold configurável
  • Screenshots em afterTest — diagnóstico automático em falha
  • npm run test:api primeiro — valida TestFlow sem emulador
  • npm run appium:doctor — valida drivers antes do primeiro run mobile
  • Um cenário por it() — falhas fáceis de diagnosticar no Allure
  • visitWithToken no smoke — token compartilhado via before
  • Fixtures JSON em fixtures/ — credenciais válidas/inválidas determinísticas
